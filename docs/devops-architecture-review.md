# FamilyShield — DevOps & Security Review

Date: 2026-04-21
Author: Platform DevOps Architect

---

## Executive summary

- Overall architecture is sensible: single OCI ARM VM running services behind Cloudflare Tunnel, Supabase for data, Redis for queue, mitmproxy for content-ID extraction.
- Main operational risks: temporary open SSH during infra deploy, plaintext secrets written into rendered `docker-compose.yml` on VM, broad admin surfaces (Portainer/Headplane/Grafana) that rely on Cloudflare Zero Trust for protection.
- High-priority remediations: stop writing long-lived plaintext secrets to files on disk, reduce attack surface of management UIs, and harden IaC state and CI secret handling.

## Architecture observations

- Single VM model (OCI Always Free ARM) hosting everything via `docker compose` (see [iac/main.tf](iac/main.tf)). This is cost-efficient but concentrates blast radius.
- Cloudflare Tunnel + Zero Trust used for exposing admin surfaces and portal. Cloudflared is started via the infra workflow and the tunnel token is generated by Cloudflare IaC.
- Supabase is external (hosted) and used for data; server-side writes use a `service_role` key (sensitive).
- `docker-compose` is rendered and SCP'd to the VM by the infra workflow and contains many sensitive environment variables (see [iac/templates/docker-compose.yaml.tpl](iac/templates/docker-compose.yaml.tpl)).

## Key findings & concerns (by area)

1) IaC & Terraform state

- `terraform` backend is S3-compatible (OCI Object Storage) and is used in CI flows. `tofu` backend config uses `skip_*` flags in CI (`skip_region_validation`, `skip_credentials_validation`). These flags are acceptable for CI but reduce early validation; ensure CI runner environment is trusted and that state storage is encrypted and access-limited.
- Sensitive variables (e.g., `supabase_service_role_key`) are passed as `TF_VAR_*` in workflows and will be present in state if any resource references them. Avoid putting long-lived secrets directly into Terraform variables that become part of the state.
- `prevent_destroy` on the persistent data volume is a good safeguard.

Files: [iac/main.tf](iac/main.tf), [iac/variables.tf](iac/variables.tf), [iac/outputs.tf](iac/outputs.tf)

Risk: Medium → High (tfstate leakage of secrets can be catastrophic).

1) Docker Compose & secret handling

- The infra workflow renders `docker-compose.yml` on the runner by substituting secrets (e.g., `SUPABASE_SERVICE_ROLE_KEY`) then SCPs the file to the VM. This writes plaintext secrets to `/opt/familyshield/docker-compose.yml` on the VM.
- The file is necessary for `docker compose up`, but storing secrets in compose files is high risk: any user with VM access or a backup of the disk can retrieve them.

Files: [iac/templates/docker-compose.yaml.tpl](iac/templates/docker-compose.yaml.tpl), [.github/workflows/infra-dev.yml](.github/workflows/infra-dev.yml)

Risk: High

1) CI/CD workflows & secret lifecycle

- The Cloudflare setup job uses `gh secret set` with a GitHub PAT to dynamically update repository secrets (`CF_ACCESS_CLIENT_ID`, `CF_ACCESS_CLIENT_SECRET`). This is convenient but:
  - Requires the PAT to be tightly scoped and audited.
  - Sets repo-level secrets that give the workflow broad access if misused.
- Workflows temporarily open SSH (admin_ssh_cidrs defaults to `0.0.0.0/0`) and only tighten after successful deploy. This creates a time-window with wide exposure.
- Some outputs (tunnel token) are written to `GITHUB_OUTPUT` and masked via `::add-mask::`, which is good; but make sure no `echo` to unmasked logs occurs.

Files: [.github/workflows/infra-dev.yml](.github/workflows/infra-dev.yml), [scripts/cloudflare-api.sh](scripts/cloudflare-api.sh)

Risk: Medium

1) Cloudflare automation

- `scripts/cloudflare-api.sh` uses `jq` to build JSON and `base64 | tr -d '\n'` for porting across platforms — this is a safe approach and addresses prior JSON/newline issues.
- Access apps creation avoids tags (good). Diagnostic logging is sent to stderr which avoids polluting command substitutions.

File: [scripts/cloudflare-api.sh](scripts/cloudflare-api.sh)

Risk: Low (implementation is careful and robust)

1) Admin UIs & attack surface

- Portainer, Grafana, Headplane, Headscale admin surfaces exist on the VM and are exposed via Cloudflare Zero Trust. Portainer binds `docker.sock` and allows full container exec — a high privilege surface. Ensure Zero Trust policies are strictly enforced and that admin accounts use 2FA.
- Caddy is used to proxy directly to Headscale on port 443 to preserve WebSocket/Upgrade headers (correct design). Ensure SSL/TLS certs and proxying rules are correct.

Files: [iac/templates/docker-compose.yaml.tpl](iac/templates/docker-compose.yaml.tpl)

Risk: High for Portainer if Zero Trust misconfigured.

1) Application code & data flow

- API uses server-side Supabase client created with `SUPABASE_SERVICE_ROLE_KEY` when available ([apps/api/src/lib/supabase.ts](apps/api/src/lib/supabase.ts)). Good pattern but must ensure service role is never exposed to browser.
- Portal uses `NEXT_PUBLIC_*` only for browser-side clients and `getSupabase()` in client components (see [apps/portal/components/AlertTable.tsx](apps/portal/components/AlertTable.tsx)). This matches the documented pattern and Playwright testing approach.

Files: [apps/api/src/lib/supabase.ts](apps/api/src/lib/supabase.ts), [apps/portal/components/AlertTable.tsx](apps/portal/components/AlertTable.tsx)

Risk: Low → Medium (if CI writes service role into client build or if compose exposes env to portal container)

1) mitmproxy & privacy/ops caution

- Transparent interception uses `iptables` REDIRECT (100.64.0.0/10 → ports 8888/8889). This must be documented for legal/consent reasons. Make sure operators and parent users are aware and consent to SSL inspection.

Files: [.github/workflows/infra-dev.yml (bootstrap SSH steps)](.github/workflows/infra-dev.yml)

Risk: Operational/legal — policy risk (not a code vulnerability but important).

## Recommendations & Remediation (prioritized)

High priority (fix within days):

1. Avoid writing long-lived plaintext secrets to disk on the VM
   - Replace rendering secrets into `docker-compose.yml` with one of:
     - Use Docker secrets (swarm or file-backed) or pass secrets at container runtime via `docker run --env-file` created only at container start with strict permissions (mode 600 + owner root), then remove the env file after startup.
     - Use an agent on the VM that fetches secrets at boot from a secret store (GitHub Actions `repo_secret` is not ideal for runtime). Consider HashiCorp Vault, AWS Secrets Manager, or OCI Vault if available.
   - Minimal quick fix: restrict `/opt/familyshield/docker-compose.yml` to `root:root` and `600` permissions immediately after write, and rotate keys more frequently.
   - Files: update [.github/workflows/infra-dev.yml](.github/workflows/infra-dev.yml) to avoid substituting service_role directly into the template; instead, write a small env file with 600 perms and/or use Docker secrets.

2. Reduce SSH blast window
   - Avoid setting `admin_ssh_cidrs = ["0.0.0.0/0"]` as a default during deploy. Instead:
     - Use ephemeral bastion or Cloudflare Tunnel SSH for the deploy step.
     - If public SSH must be used for bootstrap, require a short-lived ephemeral key (generated in workflow, revoked after use) or require admin to provide the current runner IP as an input to the workflow.
   - Files: modify [iac/variables.tf](iac/variables.tf) and workflows to require explicit `TF_VAR_admin_ssh_cidrs` when running.

3. Lock down Portainer and admin UIs
   - Enforce Cloudflare Zero Trust policies for all admin Hostnames and remove public DNS records that expose these UI endpoints without Zero Trust.
   - Consider disabling Portainer in prod or using a read-only management interface; if needed, require a separate admin network or MFA-protected long-lived accounts.

4. Remove secrets from Terraform state
   - Avoid passing `supabase_service_role_key` and other sensitive tokens as Terraform variables. Instead, pass them via CI-runner environment to remote-exec steps that write env files to VM (with secure permissions) or use provider-specific secret resources.
   - If tfvars must be used, ensure state bucket has server-side encryption, strict ACLs, and that tfstate is never exported to third parties.

Medium priority (fix within weeks):

1. Harden CI token usage and rotations
   - Audit `GH_PAT` used for `gh secret set`. Scope that PAT to minimal repo secrets write permissions and rotate regularly.
   - Restrict the lifetime of tokens and use OIDC where possible.
   - Implement GitHub Actions job-level secret access and avoid storing additional secrets unless strictly needed.

2. Improve Terraform validation in CI
   - Remove `skip_credentials_validation` and friends where possible, or run `tofu validate` in a separate pre-check job with limited credentials to detect syntax errors early.

3. Container runtime hardening
   - Avoid bind-mounting `/var/run/docker.sock` for Portainer unless strictly required. Use Portainer Agent or admin-only network.
   - Ensure containers run with non-root users where possible, set `read_only` where applicable, and apply resource limits.

Low priority / operational

1. Document privacy/consent for mitmproxy and keep opt-out procedures visible for parents.
2. Add automated secret scanning in PR checks (already present?), and ensure playbook for response to secret leak exists.

## Quick wins (can be implemented today)

- Ensure `/opt/familyshield/docker-compose.yml` is created with `chmod 600` and owned by `root:root` immediately after SCP in the workflow (already partly done — verify). See [infra-dev.yml](.github/workflows/infra-dev.yml) around the docker-compose write step.
- Add `::add-mask::` for all secret outputs and validate no secret is printed in logs. Ensure `cloudflare-api.sh` does not write secrets to stdout (it uses stderr for logs — good).
- Limit GH PAT scope used for `gh secret set` to the smallest necessary scope and rotate it.

## Monitoring, detection & alerts

- Add Prometheus/Grafana alerts for: high CPU, high memory, container restarts, Redis queue growth, failed Supabase inserts, and cloudflared not registered.
- Audit logs: enable Cloudflare Access logs and consolidate VM auth logs into a central location (e.g., Grafana Loki or external SIEM) and alert on unexpected login sources.

## Proposed next steps

1. Agree on secret handling pattern (Docker secrets vs Vault) and change infra workflow accordingly.
2. Implement ephemeral SSH access or Cloudflare Tunnel SSH for bootstrapping and remove `0.0.0.0/0` window.
3. Harden Portainer and admin surfaces (Zero Trust rule audit).
4. Schedule a run-book for incident response (secret rotation, VM forensic snapshot) and store it in `docs/deployment-operations/`.
